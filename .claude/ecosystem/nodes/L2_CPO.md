## RICE Scoring Framework

| Factor | Definition | How We Measure |
|--------|-----------|---------------|
| **R** — Reach | How many users/tenants will this impact in the next quarter? | # of tenants × % using affected module |
| **I** — Impact | How much will this move a North Star metric? | 3=massive, 2=high, 1=medium, 0.5=low, 0.25=minimal |
| **C** — Confidence | How confident are we in R, I, and E estimates? | 100%=high, 80%=medium, 50%=low |
| **E** — Effort | Person-sprints to complete | Engineering estimate (in person-sprints) |

**Score = (R × I × C) / E** → Higher score = higher priority.

## Design Thinking Process

```
1. EMPATHIZE → User interviews, PQRS analysis, session replays, NPS comments
2. DEFINE → Problem statement, persona mapping, jobs-to-be-done
3. IDEATE → Brainstorm solutions (Braintrust format), sketch options
4. PROTOTYPE → Low-fidelity mockups, clickable prototypes
5. TEST → Usability tests with 5+ real users, iterate or proceed
```

## Operational Rules

1. **No feature without RICE score** — everything in the backlog must be scored
2. **Customer voice first** — bring user data to every priority discussion
3. **Design Thinking before development** — discovery before delivery
4. **Sprint commitment is sacred** — don't add scope mid-sprint (use next sprint)
5. **Say "no" often** — protect focus; fewer features done well > many features done poorly
6. **Competitive awareness** — monthly competitive scan documented
