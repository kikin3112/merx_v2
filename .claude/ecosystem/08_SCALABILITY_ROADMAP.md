# 08 — Scalability Roadmap

> Growth phases, triggers, headcount projections, and structural evolution for chandelierp.

---

## Growth Phases

### Phase 0 — Founding (Current: 1-5 tenants)

**Trigger**: Building MVP, validating core modules.

| Attribute | Value |
|-----------|-------|
| Headcount | 1-3 (founders wearing multiple hats) |
| Squads | 1 combined squad (everyone works everything) |
| Guilds | None — standards set by founder |
| Governance | CEO decides everything, minimal process |
| Budget | $0 — all free tiers |
| Focus | Core module completion, first paying customer |

**Roles Active**: CEO/CTO (all hats), 1-2 full-stack devs

---

### Phase 1 — Validation (10-20 tenants)

**Trigger**: 10 active tenants achieved, consistent weekly signups.

| Attribute | Value |
|-----------|-------|
| Headcount | 5-8 |
| Squads | 2-3 combined squads |
| Guilds | Informal (shared standards doc) |
| Governance | Product + Architecture committees start |
| Budget | Minimal — paid tiers on critical infra only |
| Focus | Product-market fit, PLG funnel optimization |

**New Roles**:
- Product Marketing Manager (content + SEO + community)
- Dedicated QA Engineer
- DevOps Engineer (dedicated)

**Structural Change**: Separate product from engineering leadership.

---

### Phase 2 — Growth (20-50 tenants)

**Trigger**: 20 active tenants, organic growth > 50%, first enterprise inquiry.

| Attribute | Value |
|-----------|-------|
| Headcount | 12-18 |
| Squads | 4 dedicated squads |
| Guilds | Architecture + DevOps guilds formed |
| Governance | All 4 committees active |
| Budget | Revenue covers infrastructure costs |
| Focus | Module depth, integrations, DIAN compliance |

**New Roles / Structures**:
- Squad: Integrations (APIs, DIAN electronic billing, payment gateways)
- Head of Customer Support
- Second QA Engineer
- UX Designer (dedicated)

**Structural Change**: Full Spotify model activated. Chapters formalized within engineering.

---

### Phase 3 — Scale (50-200 tenants)

**Trigger**: 50 active tenants, LATAM pilot demand, support volume requires tiered system.

| Attribute | Value |
|-----------|-------|
| Headcount | 20-30 |
| Squads | 5-6 squads |
| Guilds | All 5 guilds fully staffed |
| Governance | Full committee structure + quarterly board reviews |
| Budget | Revenue-funded, optimization focus |
| Focus | Internationalization, platform reliability, enterprise features |

**New Roles / Structures**:
- Squad: Internationalization (i18n, multi-currency, multi-tax-regime)
- Dedicated Data Engineer
- SRE with on-call rotation
- Technical Writer (dedicated)
- Support N1 + N2 separated

**Structural Change**: VP roles fully separated. COO role becomes full-time.

---

### Phase 4 — Enterprise (200+ tenants)

**Trigger**: 200 tenants, enterprise contracts, compliance requirements.

| Attribute | Value |
|-----------|-------|
| Headcount | 30-45 |
| Squads | 7+ squads |
| Guilds | Guilds become Chapters with elected leads |
| Governance | Board of advisors, compliance officer |
| Budget | Series A or revenue-sustainable |
| Focus | AI/ML features, SOC2, enterprise sales |

**New Roles / Structures**:
- Squad: AI/ML (demand forecasting, anomaly detection, smart suggestions)
- VP of Revenue (splits from COO)
- Legal / Compliance Officer
- Security: SOC2 preparation team
- Customer Success Managers (per-account)

**Structural Change**: Matrix organization formalized. Dual reporting for some roles.

---

### Phase 5 — Platform (500+ tenants)

**Trigger**: Marketplace demand, partner ecosystem requests, developer API interest.

| Attribute | Value |
|-----------|-------|
| Headcount | 45-70+ |
| Squads | 8-10 squads + partner teams |
| Guilds | Cross-company guilds with partners |
| Governance | Full corporate governance |
| Budget | Growth-funded |
| Focus | Ecosystem, marketplace, API partnerships |

**New Roles / Structures**:
- Ecosystem Manager
- Developer Relations (DevRel)
- API Partnerships Manager
- SDK Team
- Partner Success Team

**Structural Change**: Platform team emerges as separate pillar. External developer ecosystem.

## Scaling Principles

1. **Add structure only when pain demands it** — don't over-engineer the org
2. **Promote from within first** — institutional knowledge matters
3. **Every new role must justify its existence** — what problem does it solve that can't be solved otherwise?
4. **Budget follows revenue** — grow the team only as revenue permits ($0 philosophy)
5. **Document before you hire** — if the process isn't documented, hiring won't fix it
