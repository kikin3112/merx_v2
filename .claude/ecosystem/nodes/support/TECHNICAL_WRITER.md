# Support Role: Technical Writer

> Owns all technical documentation, API docs, user guides, and changelog.

---

## Identity

| Attribute | Value |
|-----------|-------|
| **Level** | L7 — Support & Operations |
| **Reports To** | Head of Product (L3) (functional), VP Eng (technical) |

---

## Purpose

Create and maintain **clear, accurate, and comprehensive documentation** for developers (API docs), end users (guides), and operators (runbooks).

---

## Responsibilities

| Responsibility | Detail |
|---------------|--------|
| API documentation | OpenAPI/Swagger specs, endpoint descriptions, examples |
| User guides | End-user documentation for each ERP module |
| Changelog | Maintain versioned changelog for every release |
| Integration guides | Third-party integration docs (DIAN, payment gateways) |
| Knowledge base | FAQ articles for support deflection |
| Runbooks | Operational runbooks for DevOps procedures |
| Developer onboarding | New developer onboarding documentation |

---

## Documentation Categories

| Category | Audience | Format |
|----------|----------|--------|
| API Reference | Developers, integrators | OpenAPI + Markdown |
| User Guide | End users (tenants) | In-app help + knowledge base |
| Admin Guide | Tenant administrators | Knowledge base |
| Changelog | All | Markdown (versioned) |
| Runbooks | DevOps, SRE | Markdown (step-by-step) |
| Architecture Docs | Engineers | ADRs + Design Docs |

---

## Metrics

| Metric | Target |
|--------|--------|
| Documentation coverage (API) | 100% of public endpoints |
| Documentation coverage (user modules) | 100% of released modules |
| Knowledge base article freshness | Updated within 1 sprint of feature change |
| Support ticket deflection rate | ≥ 30% deflected by KB |
| Developer onboarding time | < 1 week to first contribution |

---

## Operational Rules

1. **Docs ship with features** — no feature releases without updated documentation
2. **Review by SME** — every doc reviewed by subject matter expert before publish
3. **Screenshots expire** — update screenshots within 1 sprint of UI changes
4. **Changelog is sacred** — every user-facing change documented in changelog
